    • 风格 [待校准@238]

为了获得外观一致的产品，我们都将遵循每种语言的外部 (如果可能) 定义的风格指南。对于其他事情，如包布局或文档布局，我们需要提出我们自己的指导方针，借鉴现在使用的当前流行风格。 [待校准@232]

此外，在可能的情况下，开发人员应使用集成工具来允许他们检查编辑器中是否遵循了这些准则。例如，每个人都应该在其编辑器中内置一个PEP8检查器，以减少与样式相关的审核迭代。 [待校准@233]

此外，在可能的情况下，包应检查样式，作为其单元测试的一部分，以帮助自动检测样式问题 (见 ament_lint_auto )。 [待校准@234]

C [待校准@235]

标准 [待校准@236]

我们将瞄准c99。 [待校准@237]

风格 [待校准@238]

我们将使用 Python's PEP7 作为我们的C风格指南，并进行一些修改和补充: [待校准@239]

  • 我们将以C99为目标，因为我们不需要支持C89 (正如PEP7所建议的那样) [待校准@240]

    • 基本原理: 除此之外，它允许我们使用 // 和 /* */ 两种风格的评论 [待校准@241]

    • 基本原理: C99现在几乎无处不在 [待校准@242]

  • C + + 风格的 // 评论是允许的 [待校准@243]

  • 始终将文字放在比较运算符的左侧，例如 0 == ret 而不是 ret == 0 [待校准@244]

    • 基本原理: ret == 0 太容易意外变成 ret = 0 [待校准@245]

以下所有修改仅在我们不编写Python模块时适用: [待校准@246]

  • 不要用 Py_ 作为所有东西的前缀 [待校准@247]

    • 而是使用CamelCase版本的包名称或其他适当的前缀 [待校准@248]

  • 关于文件string的内容不适用 [待校准@249]

我们可以使用 pep7 python模块进行样式检查。编辑器集成似乎很小，我们可能需要更详细地研究C的自动检查。 [待校准@250]

C++ [待校准@251]

标准 [待校准@236]

[需手动修复的语法]Foxy目标是c14 14。 [待校准@252]

风格 [待校准@238]

我们将使用 Google C++ Style Guide ，并做一些修改: [待校准@253]

线长 [待校准@254]

  • 我们的最大行长度是100个字符。 [待校准@255]

文件扩展名 [待校准@256]

  • 头文件应使用。hpp扩展。 [待校准@257]

    • 基本原理: 允许工具确定文件、C ++ 或C的内容。 [待校准@258]

  • 实现文件应使用。cpp扩展。 [待校准@259]

    • 基本原理: 允许工具确定文件、C ++ 或C的内容。 [待校准@258]

变量命名 [待校准@260]

  • 对于全局变量，使用带有以 g_ 为前缀的下划线的小写字母。 [待校准@261]

    • 基本原理: 在整个项目中保持变量命名案例的一致性 [待校准@262]

    • 基本原理: 易于一目了然地说出变量的范围 [待校准@263]

    • 跨语言的一致性 [待校准@264]

函数和方法命名 [待校准@265]

  • 谷歌风格指南说 CamelCase ，但是c + + 标准图书馆的 snake_case 风格也是允许的 [待校准@266]

    • 基本原理: ROS 2核心包目前使用 snake_case [待校准@267]

      • 原因: 要么是历史疏忽，要么是个人偏好，没有被linter检查 [待校准@268]

      • 不改变的原因: 追溯改变太具有破坏性了 [待校准@269]

    • 其他注意事项: [待校准@270]

      • [需手动修复的语法]``cpplint.py`` does没有检查这个案例 (除了审查之外很难执行) [待校准@271]

      • [需手动修复的语法]``snake_case`` can导致语言间更加一致 [待校准@272]

    • 具体指导: [待校准@273]

      • 对于现有项目，更喜欢现有样式 [待校准@274]

      • 对于新项目，两者都可以接受，但建议优先匹配相关的现有项目 [待校准@275]

      • 最终决定始终是开发商的自由裁量权 [待校准@276]

        • 特殊情况，如函数指针、可调用类型等。可能需要弯曲规则 [待校准@277]

      • 请注意，默认情况下，类仍应使用 CamelCase [待校准@278]

访问控制 [待校准@279]

  • 所有类成员的删除要求是私有的，因此需要访问器 [待校准@280]

    • 基本原理: 这对用户API设计过于限制 [待校准@281]

    • 我们应该更喜欢私人成员，只有在需要时才公开他们 [待校准@282]

    • 在选择允许直接成员访问之前，我们应该考虑使用访问器 [待校准@283]

    • 我们应该有充分的理由允许直接会员访问，而不是因为这对我们来说很方便 [待校准@284]

例外 [待校准@285]

  • 允许例外 [待校准@286]

    • 基本原理: 这是一个新的代码库，因此旧参数不适用于我们 [待校准@287]

    • 基本原理: 对于面向用户的API来说，有例外更惯用的C++ [待校准@288]

    • 应明确避免析构函数中的异常 [待校准@289]

  • 如果我们打算将生成的API包装在C中，我们应该考虑避免异常。 [待校准@290]

    • 基本原理: 这将使包装在C中更容易 [待校准@291]

    • 基本原理: 我们打算用C包装的代码中的大多数依赖项无论如何都不使用异常 [待校准@292]

类似功能的对象 [待校准@293]

  • 对Lambda或 “std::function `` or `` std:: bind'” 没有限制 [待校准@294]

提升 [待校准@295]

  • 除非绝对要求，否则应避免升压。 [待校准@296]

评论和文档评论 [待校准@297]

  • 将 /// 和 /** */ 注释用于 * 文档 * 目的，将 // 风格注释用于注释和一般注释 [待校准@298]

    • 类和函数注释应使用 /// 和 /** */ 风格注释 [待校准@299]

    • 基本原理: 推荐用于C/C ++ 中的Doxygen和狮身人面像 [待校准@300]

    • 基本原理: 混合 /* */ 和 // 便于块注释包含注释的代码 [待校准@301]

    • 对代码如何工作的描述或类和函数中的注释应使用 // 式注释 [待校准@302]

指针语法对齐 [待校准@303]

  • 使用 “碳 * c; `` instead of `` 碳 * c; `` or `` 碳 * c; `` because of this scenario `` 碳 * c，* d，* e;” [待校准@304]

类隐私关键字 [待校准@305]

  • 不要在 “公共:” 、 “私人:” 或 “受保护:” 之前放1个空格，所有缩进都是2的倍数更一致 [待校准@306]

    • 基本原理: 大多数编辑不喜欢不是 (软) 标签大小倍数的缩进 [待校准@307]

    • 在 “公共:” 、 “私人:” 或 “受保护:” 或2个空格之前使用零空格 [待校准@308]

    • 如果之前使用2个空格，则通过2个其他空格缩进其他类语句 [待校准@309]

    • 更喜欢零空间，即与类在同一列中的 “公共:” 、 “私有:” 或 “受保护:” [待校准@310]

嵌套模板 [待校准@311]

  • 从不向嵌套模板添加空格 [待校准@312]

    • 更喜欢 set<list<string>> (c11 11特征) 而不是 set<list<string> > 或 set< list<string> > [待校准@313]

始终使用大括号 [待校准@314]

  • 始终在 if 、 else 、 do 、 while 和 for 之后使用支架，即使身体是一条线。 [待校准@315]

    • 基本原理: 由于在体内使用宏，视觉模糊和并发症的机会更少 [待校准@316]

项目完结，系统自动填充内容 [待校准@317]

  • 使用左大括号的 function ， class ， struct 定义，但拥抱撑 if ， else ， while ， for 等 .. [待校准@318]

    • 例外: 当 if (或 while 等) 条件足够长，需要换行时，使用开放式支架 (即，不要拥抱)。 [待校准@319]

  • 当函数调用不能适合一行时，用打开的圆括号 (不在参数之间) 换行，并在下一行用2个空格缩进开始它们。继续在后续行上使用2空格缩进，以获取更多参数。(注意 Google style guide 在这一点上是内部矛盾的。) [待校准@320]

    • [需手动修复的语法] if (和 while 等) 条件也是如此，它们太长而不能放在一条线上。 [待校准@321]

CMake [待校准@350]

由于没有现有的CMake风格指南，我们将定义我们自己的: [待校准@352]

  • 使用小写命令名称 ( find_package ，而不是 FIND_PACKAGE )。 [待校准@353]

  • 使用宏时，用 _ 或合理的前缀对局部变量进行前缀。 [待校准@360]

降价/重组文本/文档块 [待校准@361]

风格 [待校准@238]

以下格式化文本的规则旨在提高可读性和版本控制。 [待校准@362]

  • [.md, .rst only] Each section title should be preceded by one empty line and succeeded by one empty line.

    • 基本原理: 在筛选文件时，它加快了对结构的概述。 [待校准@364]

  • [.rst only] In reStructured Text the headings should follow the hierarchy described in the Sphinx style guide:

    • [需手动修复的语法]``#`` with过线 (只有一次，用于文件标题) [待校准@366]

    • * with过线 [待校准@367]

    • = [待校准@368]

    • - [待校准@369]

    • ^ [待校准@370]

    • " [待校准@371]

    • 基本原理: 在筛选文档时，一致的层次结构加快了对嵌套级别的了解。 [待校准@372]

  • [.md only] In Markdown the headings should follow the ATX-style described in the Markdown syntax documentation

    • ATX样式的标头在行首使用1-6个哈希字符 (“#”) 表示标头级别1-6。 [待校准@374]

    • 应该在散列和标题标题之间使用一个空格 (例如 “# 标题1”)，以便更容易在视觉上分离它们。 [待校准@375]

    • ATX风格偏好的理由来自 Google Markdown style guide [待校准@376]

    • 基本原理: ATX风格的标题更易于搜索和维护，并使前两个标题级别与其他级别一致。 [待校准@377]

  • [any] Each sentence must start on a new line.

    • 基本原理: 对于较长的段落，开头的单个更改会使差异不可读，因为它会延续到整个段落。 [待校准@379]

  • [any] Each sentence can optionally be wrapped to keep each line short.

  • [any] The lines should not have any trailing white spaces.

  • [.md, .rst only] A code block must be preceded and succeeded by an empty line.

    • 基本原理: 空格仅在围栏代码块之前和之后直接有效。追踪这些说明将确保突出显示工作正常且一致。 [待校准@383]

  • [.md, .rst only] A code block should specify a syntax (e.g. bash).
