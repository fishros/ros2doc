概述 [待校准@9718]

ROS 2中的执行管理通过执行器的概念进行了阐述。执行程序使用底层操作系统的一个或多个线程对传入的消息和事件调用订阅、计时器、服务服务器、动作服务器等。显式执行者类 (在rclcpp中的 executor.hpp 、rclpy中的 executors.py 或rclc中的 executor.h ) 比在ROS 1中的自旋机制提供了对执行管理的更多控制，尽管基本API非常相似。 [待校准@9804]

下面，我们重点介绍cclient客户端库 * rclcpp *。 [待校准@9805]

基本用途 [待校准@9806]

在最简单的情况下，主线程通过调用 “rclcpp::spin (spin)” 来处理节点的传入消息和事件，如下所示: [待校准@9807]

int main(int argc, char* argv[])
{
   // Some initialization.
   rclcpp::init(argc, argv);
   ...

   // Instantiate a node.
   rclcpp::Node::SharedPtr node = ...

   // Run the executor.
   rclcpp::spin(node);

   // Shutdown and exit.
   ...
   return 0;
}

对 spin(node) basi调用y的调用扩展为单线程执行程序的初始化和调用，这是最简单的执行程序: [待校准@9808]

rclcpp::executors::SingleThreadedExecutor executor;
executor.add_node(node);
executor.spin();

通过调用执行程序实例的 spin() ，当前线程开始向rcl和中间件层查询传入消息和其他事件，并调用相应的调用函数，直到节点关闭。为了不抵消中间件的QoS设置，传入的消息不会存储在客户端库层的队列中，而是保存在中间件中，直到被调用的back函数处理为止。(这对ROS 1来说是一个至关重要的区别。)* 等待集 * 用于通知执行程序中间件层上的可用消息，每个队列有一个二进制标志。 [待校准@9809]

单线程执行程序也被容器进程用于 components ，即在没有显式主函数的情况下创建和执行节点的所有情况下。 [待校准@9810]

执行器类型 [待校准@9811]

目前，rclcpp提供三种从共享父类派生的执行程序类型: [待校准@9812]

The Multi-Threaded Executor creates a configurable number of threads to allow for processing multiple messages or events in parallel. The Static Single-Threaded Executor optimizes the runtime costs for scanning the structure of a node in terms of subscriptions, timers, service servers, action servers, etc. It performs this scan only once when the node is added, while the other two executors regularly scan for such changes. Therefore, the Static Single-Threaded Executor should be used only with nodes that create all subscriptions, timers, etc. during initialization.

通过为每个节点调用 add_node(..) ，所有三个执行程序都可以与多个节点一起使用。 [待校准@9814]

rclcpp::Node::SharedPtr node1 = ...
rclcpp::Node::SharedPtr node2 = ...
rclcpp::Node::SharedPtr node3 = ...

rclcpp::executors::StaticSingleThreadedExecutor executor;
executor.add_node(node1);
executor.add_node(node2);
executor.add_node(node2);
executor.spin();

在上面的示例中，静态单线程执行程序的一个线程用于一起为三个节点提供服务。对于多线程执行程序，实际并行性取决于调用的返回组。 [待校准@9815]

回调组 [待校准@9816]

rclcpp允许组织组中节点的调用。这样的 * 调用返回组 * 可以由节点类的 create_callback_group 函数创建。然后，可以在创建订阅、计时器等时指定此调用后退组-例如通过订阅选项: [待校准@9817]

auto my_callback_group = create_callback_group(rclcpp::CallbackGroupType::MutuallyExclusive);

rclcpp::SubscriptionOptions options;
options.callback_group = my_callback_group;

my_subscription = create_subscription<Int32>("/topic", rclcpp::SensorDataQoS(),
                                             callback, options);

创建的所有订阅、计时器等 (不指示调用返回组) 都分配给 * 默认调用返回组 *。可以通过 “nodebaseinterface::get_default _ 调用back_group()” 查询默认调用back组。 [待校准@9818]

有两种类型的调用返回组，其中必须在实例化时指定类型: [待校准@9819]

  • Mutually exclusive: Callbacks of this group must not be executed in parallel.

  • Reentrant: Callbacks of this group may be executed in parallel.

不同调用组的回调可以始终并行执行。多线程执行程序使用其线程作为池，根据这些条件并行处理尽可能多的调用。 [待校准@9822]

自从Galactic之后，Executor基类的接口被一个新的函数 add_callback_group(..) 改进了。这允许将调用组分配给不同的执行器。通过使用操作系统调度程序配置基础线程，特定的调用返回可以优先于其他调用返回。例如，控制循环的订阅和计时器可以优先于节点的所有其他订阅和标准服务。 examples_rclcpp_cbg_executor package 提供了这一机制的例程。 [待校准@9823]

调度语义 [待校准@9824]

如果调用back的处理时间短于消息和事件发生的时间，则执行程序basi调用y以FIFO顺序处理它们。但是，如果某些调用的处理时间较长，则消息和事件将在堆栈的较低层排队。等待集机制只向执行程序报告很少的关于这些队列的信息。详细地说，它只报告某个话题是否有任何消息。执行程序使用此信息以循环方式处理消息 (包括服务和动作)，但不按FIFO顺序处理。此外，它优先考虑所有计时器事件而不是消息。以下流程图可视化了此调度语义。 [待校准@9825]

这种语义首次在 paper by Casini et al. at ECRTS 2019 中被描述。 [待校准@9826]

展望 [待校准@9827]

虽然rclcpp的三个执行器在大多数应用程序中运行良好，但仍存在一些问题，使其不适合实时应用程序，这需要定义明确的执行时间、确定性、以及对执行顺序的自定义控制。以下是其中一些问题的摘要: [待校准@9828]

  1. 复杂和混合调度语义。理想情况下，您需要定义良好的调度语义来执行正式的时序分析。 [待校准@9829]

  2. 回调可能会遭受优先级反转。较高优先级的调用可能会被较低优先级的调用阻止。 [待校准@9830]

  3. 对调用返回执行顺序没有明确的控制。 [待校准@9831]

  4. 对特定话题的触发没有内置控制。 [待校准@9832]

此外，执行程序在CPU和内存使用方面的开销相当大。静态单线程执行程序大大减少了这种开销，但对于某些应用程序来说可能还不够。 [待校准@9833]

以下开发版本已部分解决了这些问题: [待校准@9834]

  • [需手动修复的语法] rclcpp WaitSet : rclcpp的 WaitSet 类允许直接在订阅、计时器、服务服务器、动作服务器等上等待，而不是使用执行程序。它可用于实现确定性的、用户定义的处理序列，可能一起处理来自不同订阅的多个消息。 examples_rclcpp_wait_set package 提供了几个使用该用户级等待集机制的例子。 [待校准@9835]

  • [需手动修复的语法] rclc Executor : 这个来自为微型ROS开发的C客户端库 * rclc * 的执行程序为用户提供了对调用back执行顺序的细粒度控制，并允许自定义触发条件来激活调用back。此外，它实现了逻辑执行时间 (LET) 语义的思想。 [待校准@9836]
