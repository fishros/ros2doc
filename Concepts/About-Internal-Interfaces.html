关于内部ROS 2接口 [待校准@5265]

内部ROS接口是公共的C | Api | 供正在创建 | 客户端库 | 或添加新的底层中间件的开发人员使用，但不适用于典型的ROS用户。ROS | 客户端库 | 提供大多数ROS用户熟悉的面向 | Api的用户，并且可能使用多种编程语言。 [待校准@5266]

内部API架构概述 [待校准@5267]

有两个主要的内部接口: [待校准@5268]

  • ROS中间件接口 ( rmw | API |) [待校准@5269]

  • ROS客户端库接口 ( rcl | API |) [待校准@5270]

[需手动修复的语法] rmw | API | 是ROS 2软件堆栈和底层中间件实现之间的接口。用于ROS 2的底层中间件是DDS或RTPS实现，负责发现、发布和订阅机制、服务的请求-回复机制以及消息类型的序列化。 [待校准@5271]

[需手动修复的语法] rcl | API | 是一个稍高的 | API |，用于实现 | 客户端库 | 并且不直接接触中间件实现，而是通过ROS中间件接口 ( rmw | API |) 来做到这一点。文章动作。 [待校准@5272]

如图所示，这些 | API | 被堆叠，使得典型的ROS用户将使用 | 客户端库 | | API |，e。g. ``rclcpp`` ，实现他们的代码 (可执行或库)。| 客户端库 | 的实现，例如 rclcpp ，使用 rcl 接口，该接口提供对ROS图和图事件的访问。 rcl 实施又使用 rmw | API | 访问ROS图。 rcl 实现的目的是为更复杂的ROS概念和实用程序提供一个通用的实现，这些概念和实用程序可能被各种 | 客户端库 | 使用，同时与正在使用的底层中间件保持不可知。 rmw 接口的目的是获取支持ROS客户端库所需的绝对最小中间件功能。最后， rmw | API | 的实现由中间件实现特定 | 包 |，e提供。g. rmw_fastrtps_cpp ，其库是根据供应商特定的DDS接口和类型编译的。 [待校准@5274]

在上图中还有一个标有 ros_to_dds 的盒子，此框的目的是表示可能的包的类别，允许用户使用ROS等效项访问DDS供应商特定的对象和设置。文章摘要: 该动作界面的目标之一是将ROS用户空间代码与正在使用的中间件完全隔离，因此，不断变化的DDS供应商甚至中间件技术对用户代码的影响最小。然而，我们认识到，有时进入实施并手动调整设置是有用的，尽管可能会有后果。通过要求使用这些包中的一个来访问底层DDS供应商的对象，我们可以避免在正常接口中暴露供应商特定的符号和标题。通过检查包的依赖关系以查看是否使用了这些 ros_to_dds 包中的一个，也可以很容易地看到哪些代码可能违反了供应商的可移植性。 [待校准@5275]

特定类型接口 [待校准@5276]

一直以来，| Api | 的某些部分必然特定于要交换的消息类型，例如发布消息或订阅话题，因此，需要为每种消息类型生成代码。下图布局了从用户定义的 rosidl 文件 (例如 .msg 文件) 到用户和系统用于执行特定类型功能的特定类型代码的路径: [待校准@5277]

图: 流程图 "static" 类型支持生成，从 rosidl 文件用户面临码。 [待校准@5279]

图表的右侧显示了 .msg 文件是如何直接传递给特定语言代码生成器的，例如 rosidl_generator_cpp 或 rosidl_generator_py 。这些生成器负责创建用户将包括 (或导入) 的代码，并将其用作 .msg 文件中定义的消息的内存表示。例如，考虑消息 std_msgs/String ，用户可以在C++ 中使用此文件和语句 #include <std_msgs/msg/string.hpp> ，或者他们可以在Python中使用语句 from std_msgs.msg import String 。这些语句之所以有效，是因为这些语言特定 (但与中间件无关) 生成器包生成的文件。 [待校准@5280]

另外， .msg 文件用于为每种类型生成类型支持代码。在这种情况下，类型支持是指: 特定于给定类型并由系统用于执行给定类型的特定任务的元数据或函数。对给定消息的类型支持可能包括消息中每个字段的名称和类型列表。它还可能包含对可以为该类型执行特定任务的代码的引用，例如发布消息。 [待校准@5281]

静态类型支持 [待校准@5282]

当类型支持引用代码来执行特定消息类型的特定功能时，该代码有时需要执行中间件特定的工作。例如，考虑特定类型的发布函数，当使用 "vendor A" 时，该函数需要调用 "vendor A" 的 | API |，但是当使用 "vendor B" 时，它需要调用 "vendor B" 的 | API |。为了允许中间件供应商特定代码，用户定义的 .msg 文件可能导致供应商特定代码的生成。通过类型支持文章动作，这个供应商特定的代码仍然对用户隐藏，这类似于 "Private Implementation" (或皮条客) 模式的工作方式。 [待校准@5283]

带有DDS的静态类型支持 [待校准@5284]

对于基于DDS的中间件供应商，并具体调用那些基于OMG IDL文件 ( .idl 文件) 生成代码的，用户定义的 rosidl 文件 ( .msg 文件) 被转换成等效的OMG IDL文件 ( .idl 文件)。从这些OMG IDL文件中，创建供应商特定的代码，然后在类型特定的函数中使用，这些函数由给定类型的类型支持引用。上面的图表在左手边展示了这一点，在这里 .msg 的文件被 rosidl_dds 包装消耗来产生 .idl 的文件，然后这些 .idl 给定文件语言具体DDS供应商特定类型支持生成包。 [待校准@5285]

例如，考虑Connext DDS实现，它有一个调用ed rosidl_typesupport_connext_cpp 的包。这个包负责生成代码来处理诸如发布到给定消息类型的c ++ 版本的话题之类的事情，使用 rosidl_dds 包生成的 .idl 文件来这样做。同样，该代码虽然特定于Connext，但由于类型支持中的文章摘要动作，仍然没有向用户公开。 [待校准@5286]

动态类型支持 [待校准@5287]

实现类型支持的另一种方法是具有用于发布到话题之类的通用函数，而不是为每种消息类型生成函数的版本。为了实现这一点，这个通用函数需要一些关于正在发布的消息类型的元信息，诸如字段名称和类型在消息类型中显示顺序的列表之类的内容。然后，要发布消息，您可以调用通用发布函数，并传递要发布的消息以及包含有关消息类型的必要元数据的结构。这被称为 "dynamic" 型支持，与 "static" 型支持相反， "static" 型支持需要每种类型的函数的生成版本。 [待校准@5288]

图: 流程图 "dynamic" 类型支持生成，从 rosidl 文件用户面临码。 [待校准@5290]

上图显示了从用户定义的 rosidl 文件到生成的面向用户的代码的流程。它与静态类型支持的图非常相似，并且仅在图的左侧表示的类型支持的生成方式上有所不同。在动态类型支持中， .msg 文件直接转换为面向用户的代码。 [待校准@5291]

此代码也与中间件无关，因为它仅包含有关消息的元信息。实际完成工作的函数，例如发布到一个话题，对于消息类型是通用的，并且将对中间件特定的 | Api | 进行任何必要的调用。请注意，此方法不是为dds供应商提供类型支持代码的特定包 (静态类型支持就是这种情况)，而是为每种语言提供中间件不可知包，例如g. rosidl_typesupport_introspection_c 和 rosidl_typesupport_introspection_cpp 。包名称的 introspection 部分指的是利用消息类型生成的元数据对任何消息实例进行反思的能力。这是基金ament能力，允许像 "publish to a topic" 这样的功能的通用实现。 [待校准@5292]

这种方法的优点是所有生成的代码都与中间件无关，这意味着只要它们允许动态类型支持，就可以将其重新用于不同的中间件实现。它还会导致生成的代码更少，从而减少编译时间和代码大小。 [待校准@5293]

但是，动态类型支持要求底层中间件支持类似形式的动态类型支持。对于DDS，DDS-XTypes标准允许使用元信息而不是生成的代码发布消息。为了支持动态类型支持，底层中间件中需要DDS-XTypes或类似的东西。此外，这种类型支持的方法通常比静态类型支持的替代方法慢。可以编写静态类型支持中的特定类型生成的代码，以提高效率，因为它不需要迭代消息类型的元数据来执行诸如序列化之类的操作。 [待校准@5294]
